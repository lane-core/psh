## Lessons Learned

### 1. The `_sf*.c` naming convention is inconsistent

The `_` prefix has **two different meanings** in the sfio source:

- **Thin wrappers**: `_sfdlen.c`, `_sfllen.c`, `_sfulen.c`, `_sfslen.c` —
  these just `#undef` a macro and call the `__sf_*` expansion. They provide
  non-inline function symbols for external linkage. **Zero callers in practice**
  on Darwin/arm64 where `__INLINE__` is not defined (macros used instead).

- **Real implementations**: `_sfputl.c`, `_sfputu.c`, `_sfputd.c`, `_sfputm.c` —
  these are the actual encoder implementations with full stream I/O logic.
  The macro `__sf_putl(f,v)` expands to `_sfputl(f,v)`. Deleting these
  breaks the build.

The explore agent initially classified all `_sf*.c` files as "zero-reference
wrappers" and recommended deleting them all in Phase B. The Phase B gate
(build + test) passed for the true wrappers but Phase C's build failed with
linker errors for `_sfputd` and `_sfputm`. Solution: recovered the
implementations from git and included them in sfvle.c.

**Takeaway**: Verify each file's actual content, not just its naming pattern.
The agent's grep-based reference search found "zero callers" because the
callers go through macros, not direct function calls.

### 2. New files must be `git add`ed before `just build`

The nix build copies the source tree based on git tracking. Untracked files
are excluded from the nix store copy. Phase C initially failed because
`sfvle.c` was created but not `git add`ed. The nix build silently produced
a binary without the VLE functions, leading to link errors.

**Takeaway**: Always `git add` new files before running `just build`.

### 3. Iteration path needs `just clean` after file deletions

The iteration path (local samu) uses a cached `build.ninja` generated by
`configure.sh`. After deleting source files, the old `build.ninja` still
references the deleted files' object targets. `just clean` + reconfigure
resolves this. The nix path handles this automatically (content-addressed).

**Takeaway**: A `just sync` porcelain recipe (clean + reconfigure, no build)
would streamline the deletion workflow. Not yet implemented.

### 4. Nix sandbox timing flakes are consistent

The same pre-existing pipefail/timing tests flake under the nix sandbox
across all phases. They always pass locally (`just test-one`). The flakes
are: `basic.sh` (pipefail race), `builtins.sh` (read -t pipe timing),
and the advisory tests (signal, sigchld). These are scheduling jitter
in the nix sandbox, not regressions.

### 5. N_ARRAY macro collisions in consolidation

`N_ARRAY` is `#define`d with different values in different files:
- sfdlen.c: `16*sizeof(Sfdouble_t)` (256 on 64-bit)
- _sfputl.c: `2*sizeof(Sflong_t)` (16 on 64-bit)
- _sfputu.c: `2*sizeof(Sfulong_t)` (16 on 64-bit)
- _sfputd.c: `16*sizeof(Sfdouble_t)` (256 on 64-bit)
- _sfputm.c: `2*sizeof(Sfulong_t)` (16 on 64-bit)

Solution: `#undef N_ARRAY` between sections in sfvle.c. This preserves
the original behavior while keeping the consolidation clean.

### Build Porcelain TODO

The workflow exposed a gap: after deleting/adding source files, the iteration
path needs `clean + reconfigure`. Three recipes were discussed:

| Recipe | Does | Use when |
|--------|------|----------|
| `just sync` | Clean stale objects + reconfigure (no build) | Source file set changed |
| `just rebuild` | Clean all + reconfigure + build | Want known-good iteration build |
| `just reconfigure` | (exists) Re-run configure.sh | Changed configure.sh or probes |

These are orthogonal to the sfio work and should be their own commit.
